# 巧巧小伙伴 - 产品设计文档

> 帮助 11 岁女孩健康使用平板的游戏化陪伴应用

## 1. 项目概述

### 1.1 背景

一位父亲担心 11 岁的女儿沉迷平板游戏和视频，希望通过温和、游戏化的方式帮助她建立健康的数字使用习惯。

### 1.2 目标

- 控制平板总使用时间
- 控制特定应用的使用时间
- 设置禁止使用时段（学习、睡觉时间）
- 通过游戏化激励机制，让孩子主动配合
- 提供使用数据报告，了解习惯变化

### 1.3 目标用户

| 角色 | 描述 |
|------|------|
| 主要用户 | 11 岁女孩，平板使用者 |
| 管理者 | 父亲，设置规则、查看报告 |
| 设备 | Android 小米平板 |

### 1.4 核心理念

```
信任 + 陪伴 + 激励 > 强制管控
```

以「巧巧」这个可爱的小伙伴形象陪伴孩子成长，而非以"家长监控"的姿态出现。

### 3.3 规则控制模块

#### 3.3.1 时间规则

**工作日规则（周一至周五）**
- 每日总时间：2 小时
- 禁止时段：
  - 09:00-12:00（上课时间）
  - 14:00-17:00（上课时间）
  - 21:00-07:00（睡觉时间）

**周末规则（周六、周日）**
- 每日总时间：3 小时
- 禁止时段：
  - 21:00-08:00（睡觉时间）

#### 3.3.2 应用分类规则

| 分类 | 每日限额 | 示例应用 |
|------|----------|----------|
| 🎮 游戏类 | 30 分钟 | 我的世界、王者荣耀等 |
| 📺 视频类 | 30 分钟 | 抖音、B站、爱奇艺等 |
| 📚 学习类 | 无限制 | 网课、词典、学习软件 |
| 📖 阅读类 | 无限制 | 电子书、阅读软件 |
| ❓ 未分类 | 需家长确认 | 新安装的应用 |

**应用分类策略：**

1. **预设分类数据库**：内置常见应用分类列表（约 500 个主流应用）
2. **自动匹配**：新安装应用根据包名和应用商店类别自动匹配
3. **未分类处理**：无法自动分类的应用归入"未分类"，首次使用时弹出提示让家长确认分类
4. **手动调整**：家长可随时在设置中调整任意应用的分类

#### 3.3.3 单独应用规则

可对特定 App 单独设置时间限制，覆盖分类规则。

### 3.5 积分与奖励模块

#### 3.5.1 积分获取方式

| 行为 | 获得积分 | 说明 |
|------|----------|------|
| 按时主动结束使用 | +10 阳光 | 在提醒前主动关闭 |
| 遵守每日总时间限制 | +20 阳光 | 一天没超标 |
| 遵守禁止时段规则 | +15 阳光 | 写作业/睡觉时间没偷玩 |
| 连续 3 天遵守规则 | +30 阳光 | 额外奖励 |

**积分边界条件：**

| 规则 | 说明 |
|------|------|
| 积分上限 | 9999 阳光（防止数值溢出，达到上限后不再累积） |
| 积分下限 | 0 阳光（不可为负数） |
| 兑换检查 | 积分不足时，加时券按钮置灰并显示"积分不足" |
| 积分扣减 | 仅在加时券激活时扣减，兑换页面不扣减 |

#### 3.5.2 积分消耗方式

| 兑换项 | 消耗积分 | 获得内容 |
|--------|----------|----------|
| 小加时券 | 30 阳光 | 额外 15 分钟游戏/视频时间 |
| 中加时券 | 50 阳光 | 额外 30 分钟游戏/视频时间 |
| 大加时券 | 80 阳光 | 额外 1 小时游戏/视频时间 |

#### 3.5.3 加时券机制

- **家长发放的临时加时券**：直接生效，不扣积分
- **女儿积分兑换的加时券**：从积分余额扣除，自主选择使用时间

**加时券使用规则：**

| 规则 | 说明 |
|------|------|
| 使用时段 | 仅在允许使用时段内可用，禁止时段（睡觉/上课时间）不可使用 |
| 叠加规则 | 不可叠加使用，一次只能激活一张加时券 |
| 有效期 | 积分兑换的加时券永久有效；家长发放的临时加时券有效期 7 天 |
| 使用方式 | 在使用过程中点击"使用加时券"按钮激活 |
| 退款规则 | 激活后不可退还积分 |

### 5.3.4 数据库 Schema 设计

```sql
-- 使用记录表
CREATE TABLE app_usage_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    package_name TEXT NOT NULL,           -- 应用包名
    app_name TEXT,                        -- 应用名称
    category TEXT,                        -- 分类：game/video/study/reading/other
    start_time INTEGER NOT NULL,          -- 开始时间戳
    end_time INTEGER NOT NULL,            -- 结束时间戳
    duration INTEGER NOT NULL,            -- 使用时长（秒）
    date TEXT NOT NULL                    -- 日期 YYYY-MM-DD
);

-- 规则配置表
CREATE TABLE rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_type TEXT NOT NULL,              -- 规则类型：total_time/app_category/app_single/time_block
    target TEXT,                          -- 目标：分类名/包名/时段
    weekday_limit INTEGER,                -- 工作日限额（分钟）
    weekend_limit INTEGER,                -- 周末限额（分钟）
    time_start TEXT,                      -- 禁止开始时间 HH:MM
    time_end TEXT,                        -- 禁止结束时间 HH:MM
    enabled INTEGER DEFAULT 1             -- 是否启用
);

-- 积分流水表
CREATE TABLE points_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    change INTEGER NOT NULL,              -- 积分变化（正为获得，负为消耗）
    reason TEXT NOT NULL,                 -- 原因描述
    balance_after INTEGER NOT NULL,       -- 变化后余额
    created_at INTEGER NOT NULL           -- 创建时间戳
);

-- 加时券表
CREATE TABLE coupons (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    type TEXT NOT NULL,                   -- 类型：small/medium/large
    duration INTEGER NOT NULL,            -- 时长（分钟）
    source TEXT NOT NULL,                 -- 来源：earned/parent_given
    status TEXT DEFAULT 'available',      -- 状态：available/used/expired
    expires_at INTEGER,                   -- 过期时间戳（null表示永久有效）
    used_at INTEGER,                      -- 使用时间戳
    created_at INTEGER NOT NULL           -- 创建时间戳
);

-- 每日统计表
CREATE TABLE daily_stats (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    date TEXT UNIQUE NOT NULL,            -- 日期 YYYY-MM-DD
    total_duration INTEGER DEFAULT 0,     -- 总使用时长（秒）
    game_duration INTEGER DEFAULT 0,      -- 游戏时长（秒）
    video_duration INTEGER DEFAULT 0,     -- 视频时长（秒）
    study_duration INTEGER DEFAULT 0,     -- 学习时长（秒）
    points_earned INTEGER DEFAULT 0,      -- 当日获得积分
    rules_followed INTEGER DEFAULT 1,     -- 是否遵守规则
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

-- 应用分类表
CREATE TABLE app_categories (
    package_name TEXT PRIMARY KEY,
    app_name TEXT,
    category TEXT NOT NULL,               -- 分类
    custom INTEGER DEFAULT 0,             -- 是否自定义分类
    updated_at INTEGER NOT NULL
);
```

#### 5.3.1 应用锁定方案详解

**主方案：Overlay 悬浮窗**

```
原理：当需要锁定应用时，显示一个全屏悬浮窗覆盖在目标应用上方
优点：
  - 实现相对简单
  - 兼容性较好
  - 用户授权一次即可
缺点：
  - 用户可从通知栏关闭悬浮窗
  - 部分全面屏手势可能绕过

实现步骤：
  1. 申请 SYSTEM_ALERT_WINDOW 权限
  2. 使用 WindowManager.addView() 添加全屏 View
  3. 设置 View 为不可触摸穿透
  4. 显示巧巧锁定界面
```

**备选方案：AccessibilityService**

```
触发条件：当 Overlay 方案被用户关闭时提示启用
原理：监听应用切换事件，自动跳转回锁定界面
优点：
  - 更难绕过
  - 可监听更多系统事件
缺点：
  - 需要额外授权引导
  - 部分厂商 ROM 限制较严
```

#### 5.3.2 后台保活机制

**前台服务 + 厂商适配**

```
┌─────────────────────────────────────────────┐
│           后台保活策略                        │
├─────────────────────────────────────────────┤
│                                             │
│  1. 前台服务 (Foreground Service)            │
│     - 常驻通知栏显示"巧巧守护中"              │
│     - 可折叠的最小化通知，不干扰用户          │
│     - 点击可回到主界面                       │
│                                             │
│  2. WorkManager 心跳                        │
│     - 作为备选，每 15 分钟检查一次服务状态     │
│     - 如果服务被杀死，尝试重启                │
│                                             │
│  3. 厂商 ROM 适配                           │
│     - 小米 MIUI：引导用户开启"自启动"权限     │
│     - 华为 EMUI：引导用户加入"受保护应用"    │
│     - 提供详细的设置引导页面                 │
│                                             │
│  4. 双进程守护（可选）                       │
│     - 主进程 + 守护进程互相监听               │
│     - 适用于需要更强保活的场景                │
│                                             │
└─────────────────────────────────────────────┘
```

#### 5.3.3 使用统计精度补偿

**UsageStatsManager 的限制：**
- 系统查询间隔最短 5 分钟
- 无法实时获取精确使用时间

**补偿方案：**

```
┌─────────────────────────────────────────────┐
│           精度补偿策略                        │
├─────────────────────────────────────────────┤
│                                             │
│  1. 轮询机制                                │
│     - 每 30 秒查询一次 UsageStats            │
│     - 累计使用时间，而非等待系统回调          │
│                                             │
│  2. 前台应用检测                            │
│     - 使用 UsageStatsManager.queryCurrent-  │
│       ForegroundApp() 获取当前前台应用       │
│     - 配合轮询实现近实时监控                 │
│                                             │
│  3. 提前量计算                              │
│     - 剩余 5 分钟提醒时，实际可能剩余 4-6 分钟│
│     - 最终锁定以精确时间为准                 │
│                                             │
│  4. 容错处理                                │
│     - 如果统计偏差超过 2 分钟，记录日志       │
│     - 在报告中标注"约"字提示                 │
│                                             │
└─────────────────────────────────────────────┘
```

| PACKAGE_USAGE_STATS | 获取应用使用统计 |

### 5.5 状态同步机制

使用 **Riverpod** 进行状态管理，采用响应式数据流：

```
┌─────────────────────────────────────────────────────────┐
│                    状态同步架构                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌─────────────┐    ┌─────────────┐    ┌───────────┐  │
│   │  监控模块    │───▶│  状态中心    │───▶│  首页UI   │  │
│   │ (UsageRepo) │    │(Riverpod    │    │           │  │
│   └─────────────┘    │ Providers)  │    └───────────┘  │
│                      │             │                   │
│   ┌─────────────┐    │  • 今日使用  │    ┌───────────┐  │
│   │  规则模块    │───▶│  • 剩余时间  │───▶│  提醒模块  │  │
│   │ (RuleRepo)  │    │  • 当前积分  │    │           │  │
│   └─────────────┘    │  • 规则状态  │    └───────────┘  │
│                      └─────────────┘                   │
│   ┌─────────────┐           │                         │
│   │  积分模块    │───────────┘                         │
│   │(PointsRepo) │                                     │
│   └─────────────┘                                     │
│                                                       │
│   数据流向：                                            │
│   Repository → StateNotifier → UI Consumer            │
│                                                       │
```

## 6.6 边界情况与异常处理

### 6.6.1 跨天边界处理

| 场景 | 处理方式 |
|------|----------|
| 一天的定义 | 00:00 - 23:59 为一天 |
| 跨天使用（23:55 → 00:10） | 拆分记录：23:55-00:00 归前一天，00:00-00:10 归当天 |
| 每日限额重置 | 00:00 自动重置所有分类的使用时间 |
| 禁止时段跨天 | 21:00-07:00 拆分为 21:00-23:59 和 00:00-07:00 |

### 6.6.2 异常恢复机制

| 场景 | 处理方式 |
|------|----------|
| 平板重启 | 通过 RECEIVE_BOOT_COMPLETED 自启动；读取数据库恢复状态 |
| App 崩溃 | 下次启动时检查是否有未结束的使用记录，自动修正 |
| 电量耗尽 | 使用记录实时写入数据库，重启后从数据库恢复 |
| 系统时间被修改 | 检测到时间回退时记录日志，并在报告中标记异常 |
| 数据库损坏 | 提供备份/恢复功能；损坏时重建数据库并重置 |

### 6.6.3 多用户/访客模式

| 场景 | 处理方式 |
|------|----------|
| 多用户切换 | 仅监控主用户（安装 App 的用户） |
| 访客模式 | 访客模式下的使用不计入统计 |
| 建议 | 引导家长在平板设置中禁用访客模式 |

### 6.6.4 数据完整性保障

```
┌─────────────────────────────────────────────┐
│           数据完整性策略                      │
├─────────────────────────────────────────────┤
│                                             │
│  1. 实时写入                                │
│     - 使用记录每 30 秒写入一次数据库          │
│     - 积分变化立即写入                       │
│                                             │
│  2. 校验机制                                │
│     - 关键数据写入后立即读取校验              │
│     - 发现不一致时自动修复                   │
│                                             │
│  3. 自动备份                                │
│     - 每天凌晨 3 点自动备份到本地文件         │
│     - 保留最近 7 天的备份                    │
│                                             │
│  4. 数据导出                                │
│     - 支持导出 CSV 格式使用报告               │
│     - 方便家长长期保存记录                   │
│                                             │
└─────────────────────────────────────────────┘
```
